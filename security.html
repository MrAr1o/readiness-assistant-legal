1. Overview

SteikTech takes security seriously. This policy describes how we handle security vulnerability reports for the Readiness Assistant app and what researchers and customers can expect from us.

If you believe you have found a security issue in Readiness Assistant, please report it responsibly according to this policy.

2. How to Report a Vulnerability

Email: support@steiktech.de

During the initial test and beta phase, security reports are handled through the shared SteikTech support inbox. Before public Marketplace launch, consider creating a dedicated monitored security inbox such as security@steiktech.de.

Subject line format: [SECURITY] Brief description — App: Readiness Assistant

Please include:

  • Description of the vulnerability and its potential impact
  • Steps to reproduce (as specific as possible)
  • Proof of concept if available (do not include actual exfiltrated data)
  • Your contact information (optional — for acknowledgement and follow-up)
  • Whether you would like public credit for the discovery

Please do NOT:

  • Disclose the vulnerability publicly before we have had a reasonable opportunity to assess and remediate it
  • Attempt to exploit vulnerabilities against production Confluence sites you do not own or control
  • Access or download data beyond what is necessary to demonstrate the issue
  • Attempt denial-of-service attacks, spam, or destructive testing

3. Response Commitments

MilestoneTarget timeframe
Acknowledgement of receiptWithin 48 hours of report
Initial triage and severity assessmentWithin 7 business days
Remediation timeline provided (for confirmed issues)Within 14 business days
Resolution — Critical (CVSS ≥ 9.0)Target: 14 days from confirmation
Resolution — High (CVSS 7.0–8.9)Target: 30 days from confirmation
Resolution — Medium (CVSS 4.0–6.9)Target: 60 days from confirmation
Resolution — Low/Informational (CVSS < 4.0)Addressed in next planned release

These are targets, not contractual SLAs. For a 2-person team, timelines may slip for complex issues. We will communicate proactively if a committed timeline cannot be met.

4. Severity Levels

We use a simplified severity classification based on impact and exploitability:

SeverityDescriptionExamples
CriticalRemote code execution, authentication bypass, cross-tenant data access, mass data exfiltrationForge function compromise allowing access to another tenant's data
HighSignificant data exposure, privilege escalation within an installation, stored XSS with widespread impactAdmin-level stored XSS in the app UI; leaking another user's workflow records
MediumLimited data exposure, reflected XSS, logic flaws with limited impactA user seeing another user's page title in a finding they shouldn't see
Low / InformationalMinor information disclosure, theoretical issues without practical exploit pathApp version disclosure; minor config hardening issues

5. Scope

In scope (please report):

  • Readiness Assistant app code (Forge backend resolvers and frontend)
  • Data handling vulnerabilities (e.g. raw page content being stored unintentionally)
  • Authentication or authorisation flaws within the app
  • Cross-tenant data access vulnerabilities
  • XSS, CSRF, or injection vulnerabilities in the UI
  • Sensitive data exposure in Forge logs

Out of scope (report to respective parties):

  • Atlassian Forge platform vulnerabilities → report to Atlassian: https://www.atlassian.com/trust/security/vulnerability-disclosure
  • Confluence Cloud vulnerabilities → report to Atlassian
  • Third-party npm library vulnerabilities → report to the respective project maintainers
  • Social engineering or phishing attacks

6. Triage Process

When a report is received:

  1. Receipt confirmation (48h): Automated or manual acknowledgement sent to reporter.
  2. Initial review (7 days): One of the two founders reviews the report, reproduces if possible, and assigns a severity level.
  3. Response to reporter: We inform the reporter of our assessment and planned timeline.
  4. Remediation: Depending on severity, we patch the app code and deploy an update via forge deploy.
  5. Disclosure coordination: We agree with the reporter on timing for any public disclosure.
  6. Post-fix update: We notify the reporter when a fix is deployed. For significant issues, we publish a security advisory in our release notes.

7. Responsible Disclosure Timeline

We request a 90-day coordinated disclosure window from the date of our confirmed receipt of a valid vulnerability report. This means:

  • We ask reporters not to publish details publicly for 90 days while we remediate.
  • If we resolve the issue before 90 days, we will agree with the reporter on an earlier disclosure date.
  • If we cannot resolve within 90 days, we will communicate our timeline and discuss options with the reporter.
  • For critical issues causing active harm, we may request an expedited embargo.

8. Safe Harbour

SteikTech will not pursue legal action against security researchers who:

  • act in good faith and follow this policy;
  • do not access, modify, or exfiltrate data from Confluence sites they do not own or have explicit permission to test;
  • do not disrupt service availability;
  • report findings to us before any public disclosure;
  • do not share vulnerability details with third parties before the coordinated disclosure window has elapsed.

We consider good-faith security research to be a service to the community and to our customers.

9. Recognition

We will:

  • Acknowledge responsible disclosers in our release notes (with the reporter's consent);
  • Maintain a private record of all security reports for internal tracking.

We do not currently offer a financial bug bounty programme.
